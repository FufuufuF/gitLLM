# PRD 草案（迭代式）

> 写法说明：先把“北极星”写清楚，后续每一步都在此基础上收敛功能边界与优先级。该草案以 MVP 为目标，不引入具体技术实现细节。

## 第一步：项目愿景与核心价值（The North Star）

### 1）产品一句话定义
gitLLM 是一款面向深度对话场景的「对话分支 + 语义合并」工具：当用户在追问中偏离主任务时，系统将跑题内容隔离到“探究分支”，并在结束后以“学习简报”的形式回填到主线对话，从而减少上下文污染、提升任务完成率。

### 2）核心用户画像
核心用户不是“是否会编程”，而是“是否经常进行多轮、长链路的目标导向对话”。建议先聚焦 3 类高频人群：

1. 复杂任务执行者（主力人群）
    - 典型场景：部署、排障、写方案、做数据分析、写论文、准备面试、做产品/运营策划。
    - 共性：任务主线清晰，但经常被术语解释、背景知识补齐、临时发散问题打断。

2. 学习型用户（非技术也成立）
    - 典型场景：学习新概念（Docker/财务指标/法律条款/医学名词），同时要完成一个更大的目标（交付作业/写报告/做决策）。
    - 共性：追问链条长、分支多，需要“学完再回到主线继续做事”。

3. 多方案探索者（决策与实验）
    - 典型场景：方案 A/B/C 的比较、不同实现路径的试错（例如“容器化 vs 虚拟机部署”）。
    - 共性：需要保留探索痕迹但不污染主线；失败能快速回退。

非目标用户（现阶段不优先满足）：
- 只进行一问一答、很少追问的轻度用户。
- 把聊天当作“永久知识库/笔记软件”的用户（长期记忆可作为后续扩展，而非 MVP 重点）。

### 3）成功标准（MVP）
MVP 的“成”，不是 UI 做得像 Git，而是用户在一次真实任务中能稳定完成以下闭环：

1. 主线任务可持续推进
    - 用户以“一个明确目标”开始（例如：部署后端项目）。

2. 跑题被识别并被建议隔离
    - 在用户出现概念追问/明显发散时，系统默认给出“开启探究分支”的建议（用户也可通过开关选择全手动）。

3. 分支内完成深挖
    - 用户在分支里把某个子问题追问清楚（例如：Docker 是什么、怎么安装、如何跑起来）。

4. 回到主线且上下文不被污染
    - 用户一键回到主线继续原任务，主线对话不会被迫携带分支的冗长追问全文。

5. 语义合并形成“可用的学习简报”
    - 用户选择“带回主线”，系统生成并写入一条简报（建议默认包含：学到的关键结论/操作清单/关键参数与约束/待确认事项）。

可量化验收指标（建议用于自测/内测）：
- 单次对话中，用户至少完成 1 次“建议分支 → 回主线 → 合并简报”的闭环。
- 合并后，用户能在主线继续完成后续步骤，且不需要重复解释分支中已学概念。
- 用户回到主线后 3 轮以内，模型仍能保持对主任务目标的聚焦（主观体验 + 简单打分均可）。

## 第二步：功能清单（Feature List）与优先级（MVP / Should / Could）

> 目标：把“能跑通闭环”的能力做成最小集合（MVP），其余功能按价值/成本递延。此处只写产品行为与验收口径，不写具体技术实现。

### 0）范围边界（本阶段先定清楚）

MVP 明确包含：
- 围绕“主线任务 + 探究分支 + 回主线 + 语义合并（学习简报）”的完整闭环。
- 分支支持多级嵌套，合并采用逐级合并。

MVP 明确不包含（后续再评估）：
- 团队协作/多人共享/实时协同。
- 把内容当作长期知识库的复杂能力（跨会话自动复用、全局知识图谱等）。
- 复杂的“冲突解决器”（例如自动判定哪条事实覆盖哪条事实）；MVP 只提供“提示与确认”。

### 1）功能池（完整列表，后续可扩充）

围绕「隔离 → 下钻 → 沉淀 → 复用」四步闭环，功能池按领域归类如下：

1. 对话与任务主线
    - 创建/继续一个“任务会话”（主线对话）。
    - 主线目标可见（例如顶部显示当前目标/任务标题）。

2. 分支（探究）管理
    - 创建探究分支（从当前主线节点切出）。
    - 在分支中继续对话，不影响主线上下文。
    - 分支切换（回到主线/切到任意分支节点）。
    - 多级分支（分支中再开分支）。
    - 分支关闭/丢弃（不合并直接结束）。

3. 分支建议（AI 辅助）
    - 自动建议开启分支（可开关）。
    - 用户手动开启分支（无论自动建议是否开启都可用）。
    - 建议触发的“可解释理由”（例如：当前问题属于术语解释/背景补齐/明显偏离主目标）。

4. 语义合并（学习简报）
    - 合并入口：从分支“回到主线”时提示是否带回。
    - 简报生成：将分支对话提炼为一条可注入主线的“学习简报”。
    - 简报结构（建议）：
      - 本次学到的关键结论
      - 可执行步骤/操作清单
      - 关键参数与约束（如版本、端口、路径、命令、账号权限等）
      - 待确认问题/风险点
    - 逐级合并：多级分支时必须先合并子分支到父分支，再由父分支合并回主线。

5. 可视化与可控性
    - 分支/节点概览（树状或图状均可，MVP 以“可用”为先）。
    - 当前所在位置提示（你现在在：主线/分支 A/分支 A-1）。
    - 快速回到主线。

6. 设置与数据管理
    - 自动建议开关（默认开启）。
    - 导出（会话/分支/简报）。
    - 删除会话（基础的隐私/数据清理）。

### 2）优先级分层

#### MVP（Must Have / P0）：上线必须具备

P0-1 会话主线（基本聊天）
- 用户可以创建一个“主线任务会话”，并进行连续多轮对话。

P0-2 创建探究分支（手动）
- 用户可从主线任意时刻创建“探究分支”，并为分支命名（可自动生成默认名）。

P0-3 分支切换与定位
- 用户可在主线与任意分支之间切换，且界面明确显示当前所在分支。

P0-4 逐级嵌套分支
- 用户可在分支中继续创建子分支；系统能展示层级关系。

P0-5 回到主线 + 合并提示
- 用户从分支返回主线时，系统弹出提示：
  - “仅返回主线（不带回）”
  - “生成学习简报并带回主线”

P0-6 学习简报（语义合并）写入主线
- 用户选择带回后，主线将新增一条“学习简报”消息。
- 该简报在主线后续对话中可被引用（用户能明显感知“已学过/已确认”的信息被模型记住）。

P0-7 自动建议分支（可开关，默认开启）
- 系统在检测到“明显偏离主线”的追问时，给出一次非强制建议。
- 用户可一键接受（创建分支）或忽略（继续主线）。

P0-8 分支丢弃/关闭
- 用户可关闭某个分支且不合并（系统保留或删除由产品策略决定，但对用户呈现为“结束这个探究”）。
- 该分支不合并但会被保留位历史

P0 共同验收口径（建议写进测试用例）：
- 使用“部署项目→Docker→安装→回主线继续部署”的示例，完整跑通：主线不被冗长追问污染、简报可用、后续不需要重复解释。

#### Should Have（P1）：显著提升体验，但可延后

P1-1 分支/节点概览视图增强
- 支持按时间线/树状图查看各分支与关键节点，并可一键跳转。

P1-2 简报可编辑与确认
- 合并前允许用户编辑简报（尤其是参数/版本/命令），并确认“写入主线”。

P1-3 简报中的“关键参数表”独立展示
- 将版本、端口、路径、账号、命令等以结构化形式展示，便于复核。

P1-4 建议原因解释
- 自动建议开启分支时，展示 1 句简短理由（降低被打断感）。

P1-5 合并记录与撤销
- 用户能看到“哪些简报从哪个分支合并而来”，并支持撤销一次合并（回滚简报写入）。

### 3）第二步结论（当前决策）

- MVP 聚焦：P0 功能只服务于“防污染 + 可回填”的闭环，不追求复杂知识库与协作能力。
- 最大风险点：学习简报“是否真的可用”（能否包含关键参数并被后续引用）。P1 将用“可编辑/可确认/可撤销”降低风险。

## 第三步：关键用户流程（User Flow）与关键交互

> 目标：把 MVP 的用户旅程写成“可照着做出来”的交互说明。该部分以 Web 端为前提，默认“自动建议分支=开启”，但永远允许用户手动控制。

### 1）核心信息架构（用户看到的 3 个区域）

为了让非技术用户也能理解“分支”，建议界面稳定保持 3 区域布局：

1. 左侧：结构区（会话/分支树）
    - 展示：主线（主任务）以及其下的探究分支（可嵌套）。
    - 每个分支有状态标签：进行中 / 已结束（未合并）/ 已合并。

2. 中间：对话区（当前正在聊天的那条线）
    - 永远只显示“当前所在分支”的对话，不混杂其他分支内容。
    - 对话顶部显示“当前位置面包屑”：主线 > Docker 探究 > 安装问题（示例）。

3. 右侧：上下文/简报区（可折叠）
    - 展示已合并的学习简报（主线视角下尤为重要）。
    - 也可展示当前分支的“关键参数”（P1 强化项，MVP 只做简报列表即可）。

MVP 验收点：用户不需要理解 Git，也能在视觉上理解“我现在在哪条线上聊”。

### 2）主流程 A：从主线开始 → 自动建议分支 → 合并简报 → 继续主线（MVP 必须跑通）

场景示例：用户主线目标是“部署后端项目”，过程中追问 Docker。

步骤与系统行为：

1. 创建主线会话
    - 用户动作：点击“新建任务/新建会话”，输入任务标题（可选）并开始提问。
    - 系统反馈：左侧出现“主线（主任务）”，中间进入主线对话。

2. 触发“建议开启探究分支”
    - 触发条件（产品行为描述）：当用户的追问明显属于“术语解释/背景补齐/偏离当前目标的子主题”时，系统在对话区插入一条建议卡片。
    - 建议卡片文案（示例）：
      - 标题：这看起来需要单独深入一下
      - 说明：为避免干扰主线「部署后端项目」，建议开启探究分支讨论「Docker 是什么」。
      - 按钮：开启探究分支 / 继续主线
    - 默认策略（MVP 建议）：对同一段连续追问只建议一次，避免“反复打断”。

3. 用户选择“开启探究分支”
    - 用户动作：点击“开启探究分支”。
    - 系统反馈：
      - 自动创建分支（默认名如“探究：Docker”），并切换到该分支。
      - 面包屑变为：主线 > 探究：Docker。
      - 左侧树中该分支标记为“进行中”。

4. 在分支中深入追问
    - 用户动作：继续问“Docker 是什么/怎么安装/怎么跑起来”。
    - 系统反馈：所有对话只写入该分支。

5. 从分支返回主线
    - 用户动作：点击顶部“回到主线”或左侧点击主线。
    - 系统反馈：弹出合并对话框（二选一 + 可取消）：
      - 仅返回主线（不带回）
      - 生成学习简报并带回主线

6. 选择“生成学习简报并带回主线”
    - 用户动作：点击“带回主线”。
    - 系统反馈：
      - 在主线新增一条“学习简报”（系统消息样式），并在右侧简报区可见。
      - 该分支状态更新为“已合并”。

7. 继续主线任务
    - 用户动作：继续问“那我如何用 Docker 部署这个后端项目”。
    - 验收口径：模型不再把“Docker 是什么/怎么安装”当作未知前提重复展开，而是基于简报直接进入部署步骤（或在需要时只做最小补充）。

### 3）主流程 B：手动创建探究分支（MVP 也必须支持）

适用：用户不想被系统打断，或自动建议被关闭。

1. 用户动作：在对话输入框附近点击“开启探究/新建分支”。
2. 系统反馈：
    - 允许用户输入分支名称（可选），或一键使用默认名。
    - 创建后立刻切入分支。
3. 验收口径：即使关闭自动建议，用户仍能完整跑通“分支→合并简报→主线继续”。

### 4）主流程 C：多级嵌套分支与逐级合并（你已明确的产品决策）

场景示例：主线（部署）→ 探究 Docker → 探究 Docker 安装失败（子分支）。

1. 在分支中创建子分支
    - 用户动作：在“探究：Docker”中再次点击“开启探究”。
    - 系统反馈：面包屑变为：主线 > 探究：Docker > 探究：安装失败。

2. 逐级合并规则（MVP 必须强约束）
    - 用户从子分支回父分支时：提示是否生成简报并带回父分支。
    - 子分支若未先合并回父分支：不允许直接合并到主线（给出解释提示：请先合并到上一级）。

3. 验收口径
    - 用户能清晰理解“先收纳子问题，再回到上一级”的路径。
    - 最终主线只收到“顶层可用”的简报（由逐级合并自然汇总而来）。

### 5）主流程 D：丢弃/结束分支（不合并）

产品意图：允许用户“这条探究没有价值/不想带回”，同时不让用户产生“内容丢了”的焦虑。

1. 用户动作：在分支中点击“结束探究（不带回）”。
2. 系统反馈：
    - 直接返回主线（不生成简报）。
    - 左侧树保留该分支，状态标记为“已结束（未合并）”。
3. 验收口径：用户后悔时仍可点开查看对话，但主线不受其影响。

### 6）关键微交互与文案（MVP 建议统一）

1. 位置感（必须）
    - 面包屑：清晰展示主线/分支层级。
    - 状态标签：进行中 / 已合并 / 已结束（未合并）。

2. 合并弹窗（必须，且要足够“傻瓜”）
    - 标题：要把这次探究带回主线吗？
    - 解释：带回后会生成一条简报，帮助主线继续推进；不带回则不会影响主线。
    - 按钮顺序建议：带回主线（主按钮） / 不带回 / 取消

3. 自动建议（必须“轻”）
    - 建议卡片必须可关闭/可忽略。
    - 同一段连续偏离追问默认只弹一次（避免打断感）。

## 第四步：MVP（P0）需求明细与验收标准

> 目标：把 P0 功能写成“可拆解、可验收、可拒绝歧义”的需求条目。每条包含：范围、规则、异常、验收。

### 0）术语与状态（统一口径）

- 会话（Session）：用户围绕一个任务目标的对话集合，包含 1 条主线 + 若干分支。
- 主线（Mainline）：会话的默认对话线，代表“任务主目标”的推进。
- 分支/探究（Branch / Deep Dive）：从某个节点切出的子对话线，用于处理偏离主线的子问题。
- 节点（Node）：对话中的一个“关键位置”（可理解为一次消息对话后的状态点），用于从此处切分支/回到此处。
- 学习简报（Brief）：分支合并时生成的一条结构化摘要，用于写入主线并影响后续对话。

分支状态（MVP 必须支持的显示状态）：
- 进行中：当前仍可继续对话。
- 已合并：已生成简报并写入其父级（父分支或主线）。
- 已结束（未合并）：用户选择不带回/直接结束探究。

层级规则（MVP 强约束）：
- 分支可嵌套。
- 合并必须逐级：子分支只能合并到父分支，不能越级直接合并到主线。

### P0-1 会话主线（基本聊天）

范围
- 用户可新建会话并在主线上连续多轮对话。

规则
- 会话必须存在且可被重新打开（至少在当前登录态/当前设备可见）。
- 主线对话始终可继续，不因分支存在而阻塞。

异常
- 网络/服务异常：需要可重试，并明确提示“本次回复失败，未写入会话”。

验收标准
- 用户能：新建会话 → 发送 5 轮消息 → 刷新页面 → 会话仍可继续（不要求跨设备）。

### P0-2 创建探究分支（手动）

范围
- 用户可从当前所在对话线（主线或任意分支）创建新分支。

规则
- 分支创建的“起点”必须明确：默认以当前最新节点作为分支起点。
- 分支命名：
    - 用户可输入名称；
    - 若不输入，系统生成默认名（例如“探究：{主题}”或“探究 {时间}”）。
- 创建后默认切换到新分支。

异常
- 名称为空/过长/重复：允许重复但需在结构区可区分（例如显示“（2）”）；或强制不重复（二选一，MVP 建议允许重复并自动区分）。

验收标准
- 在主线点击“开启探究”后：
    - 左侧出现新分支；
    - 中间对话区切换到新分支；
    - 面包屑展示层级。

### P0-3 分支切换与定位

范围
- 用户可在主线/任意分支之间切换查看并继续对话。

规则
- 切换只改变“当前对话线”，不应把其他分支消息混入当前对话区。
- 当前位置提示必须始终可见（面包屑或醒目标识）。
- 若用户从结构区切换到一个已结束/已合并的分支：
    - 允许查看历史；
    - 是否允许继续对话：MVP 建议允许“继续对话=自动恢复为进行中”，并提示会影响其后续合并（若已合并则按新探究处理）。

异常
- 切换时有未发送的输入：提示用户“是否保留草稿”。

验收标准
- 用户在主线与分支之间来回切换 10 次，消息不会串线。

### P0-4 逐级嵌套分支

范围
- 在分支内可继续创建子分支。

规则
- 子分支创建逻辑与 P0-2 一致。
- 结构区必须能体现层级（树状缩进/父子关系标签均可）。

异常
- 层级过深导致 UI 难以阅读：MVP 可限制最大深度（例如 5 层）并给出提示。

验收标准
- 用户可创建 3 层嵌套分支，并能在结构区准确回到任意层。

### P0-5 返回主线 + 合并提示（二选一）

范围
- 用户从任意分支返回其父级（父分支或主线）时触发“合并提示”。

规则
- 弹窗必须提供三种选择：
    1）带回（生成学习简报并写入父级）
    2）不带回（直接返回父级，分支标记为已结束未合并）
    3）取消（停留在当前分支）
- 若分支内没有任何有效对话（例如只创建未发言）：
    - “带回”按钮置灰或提示“无内容可合并”。

异常
- 用户频繁切换导致重复弹窗：MVP 可增加“本次不再提示（仅本次会话）”选项（P1 也可）。

验收标准
- 从分支返回主线时，必须出现该弹窗；选择不同按钮有明确可见结果（见 P0-6/P0-8）。

### P0-6 学习简报（语义合并）写入父级

范围
- 用户选择“带回”时生成学习简报，并写入父级对话线。

规则
- 简报最小结构（MVP 必须有标题 + 要点）：
    - 标题：本次探究主题（如“Docker：安装与基本使用”）
    - 关键结论（1-3 条）
    - 可执行步骤/清单（1-5 条）
    - 关键参数与约束（若分支出现则必须提取；没有则省略）
    - 待确认事项（可选）
- 简报写入形式：作为父级对话中的一条独立消息，可被后续对话引用。
- 合并后分支状态变为“已合并”。
- 逐级合并约束：
    - 若当前分支存在未合并的子分支：不允许合并到父级，提示“请先合并/结束子分支”。

异常（事实冲突：MVP 处理策略）
- 当简报中包含与父级已存在信息冲突的“关键参数”（如 Python 3.9 vs 3.11）：
    - MVP 不做自动决策；
    - 只在简报末尾加入“可能冲突项”提示（例如“注意：父级曾提到 3.9，本分支建议 3.11，需确认”）。

验收标准
- 合并后在父级能看到新增简报，且用户继续对话时模型能利用简报信息减少重复解释。
- 存在子分支未处理时，系统阻止越级合并并给出明确指引。

### P0-7 自动建议开启分支（可开关，默认开启）

范围
- 系统对“明显偏离主线的追问”给出一次非强制建议。

规则
- 默认开启；设置中可关闭。
- 建议呈现为轻量卡片，不阻塞用户继续输入。
- 建议卡片必须包含：
    - 建议理由（1 句即可，降低困惑）
    - 两个操作：开启探究分支 / 继续主线
- 频率控制（MVP 建议）：
    - 同一段连续追问最多建议 1 次；
    - 用户明确拒绝后，短时间内不再对同一主题重复建议。

异常
- 误判（其实没跑题）：用户点“继续主线”应立即消失，不再打扰。

验收标准
- 开启自动建议：在“Docker 是什么/怎么装”等追问出现时能看到建议卡片。
- 关闭自动建议：相同场景不出现卡片，但手动建分支仍可用。

### P0-8 结束分支（不合并）

范围
- 用户可以结束探究分支并选择“不带回”。

规则
- 结束后自动返回父级。
- 分支状态变为“已结束（未合并）”。
- 分支历史可被重新打开查看。

异常
- 若该分支存在子分支：
    - 必须先结束/合并子分支，再允许结束父分支（避免悬挂）。

验收标准
- 用户结束分支后：主线不出现简报；结构区显示该分支为“已结束未合并”；重新点开仍可查看历史。
