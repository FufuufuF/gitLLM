# gitLLM

SuoFish-gitLLM 是一个面向 AI 对话产品场景的项目，尝试把“聊天”从单线问答，扩展成可管理、可分叉、可回收的对话工作流。

它关注的不只是消息收发本身，而是当一次对话不断延展之后，用户如何切换主题、探索不同思路、保留上下文，并最终把分散的信息重新整理回主线。围绕这一目标，项目构建了会话列表、流式回复、线程分支树和分支合并等核心能力，让整个交互更接近真实的 AI 产品，而不是停留在演示性质的聊天页面。

## 项目亮点

### 分支对话

- 一段会话不只有单条主线，还可以从当前线程分叉出新的子线程
- 每个分支都保留自己的上下文，适合并行尝试不同问题拆解、提示词或方案方向
- 可以随时查看线程关系、切换活跃线程、回到上层主线继续推进

### 合并分支

- 分支不是孤立存在，而是可以合并回父线程
- 合并前会生成简报预览，帮助用户先确认这个分支到底产出了什么
- 合并后，分支结论会以结构化 brief 的形式沉淀到主线中，而不是简单拼接聊天记录

### 更适合复杂任务的对话组织方式

- 适合需求拆解、方案比较、代码设计、研究推演等容易“想到一半换方向”的任务
- 避免线性聊天里上下文被频繁打断、覆盖或污染
- 让一次对话从“记录消息”升级为“管理推理路径”

### 面向 AI 工作流的后端能力

- 支持普通请求响应式聊天
- 支持 SSE 流式输出
- 支持会话、消息、线程、上下文消息的持久化
- 支持 LangGraph checkpoint，保留图执行状态

## 这个项目解决什么问题

在线性聊天产品里，用户经常会遇到几个问题：

- 一个问题想尝试两种思路，但继续追问会把原上下文冲乱
- 一条支线聊得很深，最后又很难把有价值的结果整理回主线
- 历史消息很多，但真正重要的推理节点和分支关系并不清晰

gitLLM 的目标，就是让对话天然支持“探索”和“收束”这两种行为：

- 用分支去探索不同方向
- 用合并把有效结果收束回主线

## 核心能力

- 会话管理：创建、查看、更新聊天会话
- 消息管理：保存用户消息和 AI 回复，按线程维度组织
- 分支管理：从现有线程 fork 出新线程
- 合并管理：支持 merge preview 和 merge confirm
- 导航能力：支持 breadcrumb 和线程列表
- 流式对话：支持 token 级实时输出
- 自动标题：新会话可根据首轮消息生成标题

## 技术栈

- Python 3.12
- FastAPI
- SQLAlchemy Async
- MySQL 兼容数据库
- PostgreSQL
- LangChain
- LangGraph
- langgraph-checkpoint-postgres
- ChromaDB（预留）

## 当前状态

项目目前是 MVP 版本，已经具备完整的分支对话后端雏形，但仍有一些能力处于演进中：

- 鉴权还是本地占位实现
- 模型配置目前主要通过环境变量驱动
- auth、model_configs 等模块仍保留扩展空间

## 快速运行

如果你只是想本地跑起来看看服务：

```powershell
uv sync
uv run python scripts/init_db.py
./scripts/dev.ps1
```

默认情况下，PowerShell 启动脚本监听在 9090 端口。服务启动后可访问：

- /health
- /docs

运行前需要准备基本环境变量，包括：

- DATABASE_URL
- CHECKPOINT_DATABASE_URL
- QWEN_MODEL_NAME
- QWEN_MODEL_BASE_URL
- QWEN_MODEL_API_KEY
